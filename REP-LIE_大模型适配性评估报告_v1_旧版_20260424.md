# REP-LIE 代码库大模型剪枝适配性评估报告

## 一、总体评估结论

**核心结论**：代码库对大模型（LLM）剪枝的适配性为 **部分适配，存在多处需要修改的兼容性问题**。

| 维度 | 评估结果 | 说明 |
|------|---------|------|
| 架构兼容性 | ⚠️ 基础兼容 | 支持 LLaMA/Mistral/Qwen 架构识别 |
| 推理延迟记录 | ❌ 缺失 | 仅记录训练阶段 FLOPs，无推理阶段延迟测量 |
| 资源消耗记录 | ⚠️ 部分记录 | 仅记录训练阶段内存，推理显存未记录 |
| 剪枝层名适配 | ⚠️ 部分问题 | 注意力层可适配，FFN 层仍有问题 |
| FLOPs 计算 | ⚠️ 估算值 | 仅理论公式估算，无实际推理测量 |

### 2.3 仍存在的兼容性问题

#### 问题 1：FFN 层名识别不完整

**位置**：`loraprune/utils.py` 第 176 行

```python
is_ffn = 'intermediate' in name and 'attention' not in name  # ❌ 问题所在
```

**问题说明**：LLaMA/Mistral 的 FFN 层名为 `mlp.gate_proj`、`mlp.up_proj`、`mlp.down_proj`，不包含 `intermediate` 关键字。

**影响**：LLaMA/Mistral 的 FFN 层不会被识别为剪枝目标，稳定性分数收集完全跳过 FFN 层。

**修复建议**：

```python
is_ffn = ('intermediate' in name and 'attention' not in name) \
         or last_part in ('gate_proj', 'up_proj', 'down_proj')
```

#### 问题 2：注意力层判断逻辑冗余

**位置**：`loraprune/utils.py` 第 117 行

```python
if 'self_attn' in layer_name or 'attention' in layer_name:  # 条件重复
```

该判断在 `apply_masked_modules` 函数中使用了 `or` 逻辑，但前面的判断已经有 `self_attn in name`，可能导致某些 LLaMA 层的掩码应用路径不正确。

### 6.2 建议增加的代码（推理延迟测量）

#### 增加 1：推理延迟测量工具函数

**文件**：`loraprune/utils.py`（新增）

```python
def measure_inference_metrics(model, dataset, device, num_runs=100, warmup=10):
    """测量模型推理延迟和资源消耗
    
    Args:
        model: 待测模型
        dataset: 推理数据集
        device: 设备
        num_runs: 测量次数
        warmup: 预热次数
    
    Returns:
        dict: 包含延迟和资源消耗的指标
    """
    import time
    import numpy as np
    
    model.eval()
    latencies = []
    memory_records = []
    
    with torch.no_grad():
        # 预热
        for i, batch in enumerate(dataset):
            if i >= warmup:
                break
            if isinstance(batch, dict):
                batch = {k: v.to(device) for k, v in batch.items()}
            else:
                batch = batch.to(device)
            _ = model(**batch)
            torch.cuda.synchronize()
        
        # 正式测量
        for batch in dataset:
            if len(latencies) >= num_runs:
                break
                
            # 显存记录
            torch.cuda.reset_peak_memory_stats()
            mem_before = torch.cuda.memory_allocated()
            
            # 延迟测量
            start = time.perf_counter()
            if isinstance(batch, dict):
                batch = {k: v.to(device) for k, v in batch.items()}
            else:
                batch = batch.to(device)
            _ = model(**batch)
            torch.cuda.synchronize()
            latency = time.perf_counter() - start
            
            # 记录
            latencies.append(latency * 1000)  # ms
            memory_records.append(torch.cuda.max_memory_allocated() / 1024**2)  # MB
    
    return {
        'avg_latency_ms': np.mean(latencies),
        'p50_latency_ms': np.percentile(latencies, 50),
        'p95_latency_ms': np.percentile(latencies, 95),
        'p99_latency_ms': np.percentile(latencies, 99),
        'throughput_tokens_per_sec': calculate_throughput(dataset, latencies),
        'avg_memory_mb': np.mean(memory_records),
        'peak_memory_mb': max(memory_records)
    }
```

#### 增加 2：推理评估回调

**文件**：`trainer_sb.py` 或 `main_llm.py`

```python
class InferenceMetricsCallback(TrainerCallback):
    """在训练过程中周期性测量推理性能"""
    
    def __init__(self, eval_dataset, eval_steps=1000):
        self.eval_dataset = eval_dataset
        self.eval_steps = eval_steps
        self.metrics_history = []
    
    def on_step_end(self, args, state, control, **kwargs):
        if state.global_step % self.eval_steps == 0:
            model = kwargs['model']
            metrics = measure_inference_metrics(
                model, self.eval_dataset, args.device
            )
            self.metrics_history.append({
                'step': state.global_step,
                **metrics
            })
            print(f"[推理性能] Step {state.global_step}: "
                  f"延迟={metrics['avg_latency_ms']:.2f}ms, "
                  f"显存={metrics['peak_memory_mb']:.2f}MB")
```
